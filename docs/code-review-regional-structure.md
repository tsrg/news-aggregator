# Ревью кода: структура регионального новостного сайта

**Дата:** 27.02.2025  
**Область:** frontend (Nuxt 3), backend (Express), seed, документация.

---

## Резюме

Реализация соответствует плану из `regional_news_site_structure_ee30e914.plan.md`. Код читаемый, используются Composition API и TypeScript. Исправлены несколько багов и отмечены рекомендации на будущее.

---

## Исправленные замечания

### 1. Доступность (a11y) в layout

**Было:** `aria-expanded="menuOpen"` — в DOM попадала строка `"menuOpen"`, а не значение.  
**Исправлено:** `:aria-expanded="menuOpen"` в `frontend/layouts/default.vue`. Скринридеры и тесты теперь видят корректное состояние меню.

### 2. Дублирование «Главная» в хлебных крошках

**Было:** На главной передавались `items="[{ label: 'Главная', to: '/' }]"`, компонент сам выводит «Главная» → получалось «Главная / Главная».  
**Исправлено:** На главной передаётся `items="[]"`, отображается только «Главная».

### 3. Неизвестный slug раздела

**Было:** При переходе на `/section/unknown` запрос с пустым `sectionId` не уходил, экран мог оставаться в «Загрузка...» или показывать пустой список.  
**Исправлено:** В `frontend/pages/section/[slug].vue` добавлено состояние `sectionNotFound`: после загрузки списка разделов при отсутствии раздела с таким slug выводится сообщение «Раздел не найден.»

## Рекомендации (без правок в коде)

### Безопасность: санитизация HTML на SSR

В `news/[id].vue` и `[slug].vue` контент выводится через `v-html="sanitizedBody"`. Плагин `dompurify.client.ts` вызывает DOMPurify только на клиенте (`import.meta.client`); на сервере используется только удаление `<script>`. При SSR в HTML может попасть неочищенный контент.

**Рекомендация:** санитизировать на сервере так же строго (например, `isomorphic-dompurify` или серверный аналог), либо на время не рендерить `body` в SSR и показывать его только после гидрации.

### Единый тип ответа API sections

Бэкенд отдаёт массив: `res.json(sections)`. Фронт ожидает массив: `useFetch<Section[]>(...)`. В других местах (например, `menus/header`) ответ имеет вид `{ items: [...] }`. Для единообразия и расширяемости можно ввести обёртку `{ items: sections }` в `GET /api/sections` и на фронте брать `data.value?.items` — по желанию, не обязательно.

### Типы и переиспользование

Типы `NewsItem`, `NewsResponse`, `Section` объявлены локально в `index.vue` и частично дублируются в `section/[slug].vue` и `news/[id].vue`. Имеет смысл вынести общие типы в `types/` или `types/news.ts` и импортировать — меньше расхождений и проще поддержка.

### Главная: загрузка по разделам

В `index.vue` блок «Разделы» заполняется через `watchEffect` с `Promise.all` по каждому разделу из `sectionSlugs`. При большом числе разделов будет много одновременных запросов. Можно ограничить параллелизм или запросить «последние по разделам» одним endpoint’ом на бэкенде.

### Семантика и SEO

- На главной все блоки обёрнуты в `<section>` с заголовками — хорошо.
- В `NewsCard` заголовок новости — `<h2>`; на странице статьи — `<h1>`. При большом количестве карточек на одной странице иерархия `h1` → множество `h2` корректна. При появлении единого «Главная» как `h1` стоит проверить, что внутри блоков остаётся один логический `h1` (например, только у первой карточки или у заголовка блока).
- У изображений в карточках сейчас `alt=""`. Для доступности и SEO лучше передавать осмысленный `alt` (например, от `item.title`), если нет отдельного поля описания картинки.

### Мобильное меню

Бургер и выпадающее меню работают. Закрытие по клику внутри `nav` реализовано через `@click="menuOpen = false"`. Клик по ссылке закрывает меню и переходит по маршруту — ок. Стоит убедиться, что при переходе по `NuxtLink` фокус не остаётся на бургере (при необходимости — сброс фокуса или управление им после навигации).

### Пагинация

На странице раздела пагинация по `page`/`limit` реализована, кнопки «Назад»/«Вперёд» и отключение на первой/последней странице выглядят корректно. Для очень больших `totalPages` можно добавить лимит (например, не больше 100 страниц) или перейти на «бесконечный скролл»/подгрузку — по продуктовому решению.

## Что сделано хорошо

- **Composition API + `<script setup>` + TypeScript** — единообразно во всех страницах и компонентах.
- **Фиксированный регион** — один источник правды (`useRegion()` из `runtimeConfig`), без переключателя в UI.
- **Главная как контейнер** — отдельные блоки и запросы к API, переиспользование `NewsCard`.
- **Хлебные крошки** — единый компонент, семантика `nav` и `aria-label`, корректные цепочки на главной, разделах, статье и статике.
- **Layout** — меню и футер из API, бургер и адаптив, без лишней логики.
- **Seed** — разделы, пункты меню и статические страницы соответствуют плану; при необходимости контент можно менять в админке.
- **Backend** — фильтры `region`, `section`, пагинация и лимиты выглядят согласованно с фронтом.
